## Development loop

1. An application begins from the published/closed SFT derivations in the main corpus.
2. Development occurs in the relevant standalone project.
3. If the application exposes a missing mathematical requirement, that requirement is returned to the main SFT corpus.
4. The missing derivation is created there from the One and earlier steps, receives its closure guards/tests, and is added chronologically to `OneFoldMaster.md`.
5. The closed derivation is imported into or consumed by the standalone development project.
6. Application development continues under DIRECT, FORWARD, or CONSTITUTIONAL provenance.
7. When the application mechanism or milestone is worked, verified, and established, a reviewed release is imported into the main SFT corpus.
8. The main-corpus copy becomes the published/reference snapshot; further active development continues in the standalone project.

Forcing and derivation validity are decided by the engines and corpus standards: every admitted result must trace to the One, close the required guards, and halt on violation. Neither Maria Smith nor an agent supplies a discretionary validity threshold. Maria interprets those engine-valid results, decides which conclusions are publishable, and redirects the investigation. Her approval of a conclusion for publication is the release trigger. At that point, the canonical development project, main-corpus application snapshot, related master-spine status, all related papers/READMEs, claim registry, certificates, and evidence links are synchronized as one coordinated release. Historical and voided records remain preserved but clearly superseded.

This is a feedback loop:

`main theory/reference → standalone development → missing derivation back to main → closed derivation to application → established application release back to main`.

## Current interpretation of divergence

Divergence between the main-corpus application copy and a standalone folder is expected during active development. It is not automatically drift or corruption.

The audit question is:

- Is the standalone copy active development not yet established for import?
- Is the main copy the last established published/reference release?
- Are supporting derivations being created in the main corpus and versioned back into the application?
- Has an established milestone been imported with its evidence and source identity?

Only unexplained divergence, reverse overwriting, or an import without provenance is a corpus-security defect.
